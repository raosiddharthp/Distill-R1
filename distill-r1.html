§1

Motivation // why this exists

Between 2023 and 2025, enterprises across retail, logistics, and manufacturing deployed LLM-powered assistants using proprietary APIs — OpenAI, Anthropic, Google. The pattern was uniform: connect to a hosted model, ship fast, worry about costs later. In 2026, later has arrived.

At 10 million queries per day — a modest throughput for a mid-size retailer's product assistant — GPT-4o costs roughly $50,000/day in inference alone. The CFO questions the line item. The CISO flags that customer query data is leaving the building. The ML team has no mechanism to make the model know your SKUs, your return policies, your brand voice. Three problems. One architectural decision solves all three: distil a domain-expert model you own.

Design constraint. Distill-R1 is explicitly not about distilling Claude or GPT. That violates their Terms of Service. This architecture uses Llama 3.3 70B as the teacher — an open-source model whose 3.1+ license explicitly permits using its outputs to train other models. The result is legally clean, reproducible, and auditable.

Three developments in 2024–2026 made this approach production-viable. First, QLoRA reduced VRAM requirements for fine-tuning a 7B model from ~120 GB to ~16 GB — a single consumer GPU. Second, Unsloth made QLoRA training 2× faster with 60% less memory. Third, Llama 3.3 70B achieved 88.4% on HumanEval — competitive with GPT-4o — at zero per-token cost via open weights.

Knowledge Distillation // the technical mechanism

Knowledge distillation transfers learned reasoning patterns from a large teacher model into a smaller student model. The student does not copy answers — it learns how the teacher thinks about a domain. Formally, the student minimises the KL-divergence between its output distribution and the teacher's softer, more informative distribution:

# Knowledge distillation loss (Hinton et al., 2015)
# Combined cross-entropy on hard labels + KL on soft teacher logits

L_distill = alpha * L_CE(student_logits, hard_labels)
           + (1 - alpha) * T^2 * KL(
               softmax(teacher_logits / T),
               softmax(student_logits / T)
           )

# T = temperature (2-4) · alpha = task loss weight (0.3-0.5)
# High T softens distributions, exposes richer inter-class information
# Teacher: Llama 3.3 70B · Student: Qwen2.5 7B · Method: QLoRA rank=16
Why this is legal. Meta's Llama 3.1+ license explicitly states: "we're making changes to our license, allowing developers to use the outputs from Llama models to improve other models." The Distill-R1 teacher is Llama 3.3 70B. Its synthetic outputs are legally usable as training data. No proprietary model (Claude, GPT, Gemini) is involved at any stage.
01
Synthetic Dataset Generation
Teacher (Llama 3.3 70B via OpenRouter free API) generates 500–2000 domain Q&A pairs from raw product catalogues, policy PDFs, and support ticket logs. Chain-of-thought prompting forces the teacher to expose its reasoning, not just its answers. Output: JSONL in Alpaca/ShareGPT format stored in Cloud Storage.
02
QLoRA Fine-Tuning
Student (Qwen2.5 7B) is 4-bit quantised via NF4, then fine-tuned using Low-Rank Adaptation adapters — only ~0.3% of parameters are updated. Frozen base weights handle general language; adapters learn domain reasoning. Runs on single T4 GPU (~4hrs). Output: adapter_config.json + adapter_model.bin (~80MB).
03
Evaluation & Adapter Merge
Student evaluated against held-out test set using Exact Match (EM), ROUGE-L, and LLM-as-judge scoring using the teacher itself. Passing adapters are merged into base weights for zero-latency-overhead production inference. Output: merged safetensors pushed to HF Hub private repo.

Architecture // GCP reference design

The production architecture is a two-plane design: a training plane (Vertex AI Pipelines, run once per model version) and a serving plane (Cloud Run + Vertex AI Prediction, always-on). The two planes share only the Model Registry as interface, enabling independent scaling and CI/CD.
