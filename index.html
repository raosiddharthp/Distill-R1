Distill-R1: Domain-Adapted Supervised Fine-Tuning
via Synthetic Teacher Data

A GCP Reference Architecture for Retail Product Intelligence at Scale

This document describes the design, implementation, and operational considerations for Distill-R1 — a framework for fine-tuning a small open-weight student model (Qwen2.5 7B) on domain-specific synthetic data generated by a larger open-source teacher model (Llama 3.3 70B). The primary contribution is a reproducible GCP reference architecture spanning data ingestion, synthetic dataset generation, QLoRA supervised fine-tuning, rigorous multi-metric evaluation, and production serving via Vertex AI. This document explicitly distinguishes the method used (black-box SFT on teacher completions) from full logit-level knowledge distillation, provides a first-order comparison against RAG as an architectural alternative, and includes a complete security posture, TCO model, and failure-mode analysis.

§1

Problem Statement // the three-force trap

Enterprise teams deploying LLM-powered assistants face a compounding failure mode that worsens as query volume scales. Three forces interact: API cost growth is superlinear with volume; data sovereignty risk is constant but compliance exposure accumulates; domain fit degrades as catalog size and policy complexity increase while the base model remains static.

Figure 1 — The three-force enterprise LLM trap and proposed resolution vector

The resolution of all three forces requires a single architectural commitment: own the model weights, run inference within your own network boundary, and encode domain knowledge at training time rather than prompt time. Two candidate patterns satisfy this: Retrieval-Augmented Generation (RAG) with a self-hosted base model, and Supervised Fine-Tuning (SFT) of a domain-adapted student. Section §3 addresses this choice directly before committing to the architecture.

§2

Method: SFT on Teacher Completions // corrected from v1

▲ v1 Correction — Critical
v1 described this method as "knowledge distillation" and displayed a KL-divergence loss equation requiring access to teacher logits. This was incorrect. The actual pipeline — teacher generates text completions via API, student fine-tunes on those completions — is black-box SFT on synthetic data, not logit-level KD. Hinton-style KD (2015) requires the full pre-softmax probability distribution over the vocabulary from the teacher at training time. OpenRouter's free inference API returns decoded tokens only. This section formally documents the distinction and describes both paths.

2.2 — Actual Training Objective

The implemented training objective is standard cross-entropy on teacher-generated completions. The student learns to reproduce the teacher's surface behavior on domain-specific inputs, not to match the teacher's internal probability distribution.

# Actual implemented loss: CE on teacher-generated completions # Teacher: Llama 3.3 70B via OpenRouter API (decoded text only) # Student: Qwen2.5 7B with QLoRA adapters (rank=16, alpha=32, dropout=0.05) # Target modules: q_proj, k_proj, v_proj, o_proj, gate_proj, up_proj, down_proj def sft_loss(student_logits, teacher_completion_tokens): # Standard next-token prediction on teacher-decoded text # shape: [batch, seq_len, vocab_size] -> [batch, seq_len] loss = F.cross_entropy( student_logits.view(-1, student_logits.size(-1)), teacher_completion_tokens.view(-1), ignore_index=-100 # mask prompt tokens, train on completion only ) return loss # Extension path (white-box KD) — requires self-hosted teacher # def kd_loss(student_logits, teacher_logits, T=2.0, alpha=0.4): # soft_teacher = F.softmax(teacher_logits / T, dim=-1) # soft_student = F.log_softmax(student_logits / T, dim=-1) # kl = F.kl_div(soft_student, soft_teacher, reduction='batchmean') * T**2 # ce = F.cross_entropy(student_logits, hard_labels) # return alpha * ce + (1 - alpha) * kl # Hyperparameters used in this implementation config = { "lora_r": 16, # LoRA rank "lora_alpha": 32, # scaling: effective lr = lr * (alpha/r) "lora_dropout": 0.05, "lr": 2e-4, "lr_scheduler": "cosine", "warmup_ratio": 0.03, "epochs": 3, "per_device_batch": 4, "grad_accumulation": 8, # effective batch = 32 "quant": "nf4", # double quantization enabled "max_seq_len": 2048 }

2.3 — Licensing: Qualified Statement

v1 Correction — Licensing Overstated. The Llama 3 Community License (Meta, 2024) does permit using model outputs to train derivative models. However, it contains two conditions that enterprise legal teams must evaluate: (1) if the licensee's product reaches 700 million monthly active users, a separate commercial license from Meta is required; (2) the license prohibits use of Llama model outputs to "improve the performance of other large language models" in a way that competes with Meta's commercial products — the scope of this restriction is legally ambiguous and unresolved in case law as of 2026. The characterization "legally clean" in v1 was therefore premature. This architecture recommends formal legal review before production deployment at enterprise scale. Ref: Meta Llama 3 Community License Agreement, https://llama.meta.com/llama3/license/ (2024) [6]

RAG vs. SFT: Architectural Decision // first-order comparison

▲ v1 Correction — High Severity
v1 omitted RAG entirely from its architecture comparison. For the problem described — domain grounding on a live SKU catalog with frequent updates — RAG is the dominant production pattern and must be evaluated as the primary alternative before committing to fine-tuning. This section provides that comparison and defines the conditions under which SFT is preferred.

Retrieval-Augmented Generation (Lewis et al., 2020 [7]) grounds a base model on live document stores at inference time, eliminating retraining cadence. Supervised Fine-Tuning encodes domain knowledge into model weights at training time, enabling zero-latency recall but requiring periodic retraining as the domain evolves. The choice is primarily determined by knowledge volatility and query latency budget.

Architectural decision for this implementation. The demo domain uses a stable product knowledge base (policy documents, product specs, return procedures) with a monthly refresh cycle. For this knowledge volatility profile, SFT is appropriate. The simulator scenarios (SKU shipping, return policy, spec comparison) are stable facts. The stock availability scenario ("which sofas are in stock?") is explicitly out of scope for SFT; in production, that query type routes to a RAG path or live inventory API. Section §12 (Simulator) has been updated to reflect this boundary. Ref: Lewis et al. (2020) "Retrieval-Augmented Generation for Knowledge-Intensive NLP Tasks" [7]; Gao et al. (2023) "RAG for LLMs: A Survey" [8]

Training Plane: Data Pipeline & SFT // Vertex AI Pipelines

5.1 — Dataset Quality Pipeline (New in v2)

v1 stated "500–2,000 Q&A pairs" without addressing coverage, quality filtering, or deduplication. A 7B parameter model fine-tuned on unfiltered synthetic data risks memorization of teacher artifacts, reward hacking, and poor generalization to real user queries. The following pipeline enforces quality gates before any example reaches the trainer.

Figure 6 — Dataset quality pipeline · teacher generation → filtering → deduplication → train/val/test split
01
Synthetic Generation
Teacher (Llama 3.3 70B, T=0.3 for factual stability) generates Q&A pairs from domain documents using structured CoT prompts. Output format: ShareGPT JSONL. Temperature set low to reduce teacher hallucination contaminating training data. Stored in GCS with CMEK encryption.
02
Quality Filtering + Dedup
Three-stage filter: (1) teacher refusal detection via pattern match; (2) schema validation — instruction/input/output keys present, output within 20–512 tokens; (3) MinHash LSH deduplication at Jaccard >0.8 threshold. ~23% rejection rate expected on first generation.
03
QLoRA Training + Adapter Merge
Qwen2.5 7B 4-bit NF4. LoRA rank=16, alpha=32, dropout=0.05. Target modules: all attention + MLP projections. LR=2e-4, cosine schedule, 3 epochs, effective batch=32 (grad_accum=8). Training hardware: A100 40GB on Vertex AI (~4hrs, ~$50 one-time); T4 16GB is viable on Kaggle free tier (~3–4hrs) for portfolio/dev runs. Passing adapters (eval threshold: EM>0.85 AND ROUGE-L>0.72 AND judge>4.0/5) merged via merge_and_unload() for zero-overhead inference.

Under concurrent load on a single T4 replica, vLLM's continuous batching queues requests; p99 reaches ~380ms at 50 RPS. Security Architecture // v1 gap closed

▲ v1 Correction — High Severity
v1 contained no security architecture. "SOC2-ready" appeared as a text badge. This section closes the gap with a complete security posture covering network isolation, identity, data protection, and regulatory compliance for PDPA (Singapore) and GDPR.
⚠ Deployment Anti-Pattern — Do Not Use --allow-unauthenticated on Cloud Run. This flag disables all IAM-based access control and creates a fully public endpoint, directly contradicting the VPC SC perimeter, Workload Identity configuration, and PDPA data residency posture described in this section. The Cloud Run gateway must be deployed with --no-allow-unauthenticated and accessed exclusively via IAP (end-user traffic) or Workload Identity (service-to-service). Any convenience shortcut in local development must be reverted before production deployment.
Figure 8 — Security architecture: network isolation · identity · data protection · audit
⚠ Data Sovereignty Scope: Training Pipeline vs. Serving. The sovereignty guarantee ("data never leaves GCP region") applies to the serving plane only. During training, domain documents are sent to OpenRouter (an external third-party proxy) to call the Llama 3.3 70B teacher for synthetic data generation. For retail product catalogues and FAQ documents, this is generally acceptable. However, for healthcare, financial, or legal verticals, sending source documents to any external API — even for training — may violate HIPAA, MAS TRM, or client confidentiality obligations. In those cases, self-host the teacher model on Vertex AI Training and eliminate the OpenRouter dependency entirely for the generation step. The OpenRouter API key must never be used at serve time.

Evaluation Protocol // formalized from v1

▲ v1 Correction — Critical
v1 stated "~94% accuracy" with no benchmark definition. Evaluation used LLM-as-judge where the judge was the same teacher model that generated training data — a closed-loop conflict of interest. This section defines a reproducible, multi-metric evaluation protocol with an independent judge: Llama 3.1 8B via OpenRouter free tier — a different model family from the Llama 3.3 70B teacher, free to run, and not GPT-4o (which would introduce API cost and data egress to OpenAI, contradicting the sovereignty posture). Thresholds: EM >0.85, ROUGE-L >0.72, judge >4.0/5.

8.1 — Evaluation Design Principles

Three requirements drive the evaluation design: (1) metric independence — the judge must not be the same model as the teacher; (2) test set independence — test examples must not be drawn from the same generation pipeline as training data; (3) multi-metric coverage — no single metric captures all failure modes of structured domain Q&A.

MetricWhat It MeasuresThreshold to PassFailure Mode CaughtLimitation
Exact Match (EM)Binary: predicted == reference (normalized)>0.85Completely wrong answers, hallucinated valuesPenalizes correct paraphrase; too strict for long responses
ROUGE-LLongest common subsequence overlap>0.72Partial answers, missing key factsRewards n-gram overlap regardless of factual accuracy
Llama 3.1 8B-as-judge (independent, via OpenRouter free tier)Rubric: accuracy + hallucination + completeness (0–5 scale)>4.0/5 meanSubtle hallucinations, incomplete answers, off-domain responsesRequires prompt engineering; judge bias toward verbose answers; not GPT-4o quality but free and independent of teacher
Calibration checkJudge agreement rate with EM verdict on unambiguous cases>85% agreementJudge prompt miscalibrationOnly applicable for unambiguous factual queries

8.2 — Test Set Construction

The test set (n=178) is drawn from real support ticket logs (anonymized), not the synthetic pipeline. This ensures evaluation against the actual distribution of user queries, not teacher-preferred phrasings. The test set is further partitioned by query type to expose per-category performance.

Figure 9 — Evaluation results by query category · base Qwen2.5 7B vs. Distill-R1 fine-tuned

n=178 total test examples drawn from anonymized support ticket logs (independent of training pipeline). EM scores. Stock availability marked out-of-scope (OOS) for SFT — model not trained on live inventory; this category routes to RAG path in production. 95% CI for in-scope categories: ±4.2pp (Wilson interval).

Reported accuracy range: 89–97% (in-scope categories). Weighted mean across in-scope query types = 94.8% EM. The v1 figure (~94%) was coincidentally close to the correct value but was not derived from this protocol. The stock availability category is intentionally excluded from the weighted mean — it is architecturally out of scope for fine-tuning and should not inflate or deflate the reported figure. LLM judge: Llama 3.1 8B (OpenRouter free tier) — independent of the Llama 3.3 70B teacher. Ref: Evaluation methodology follows Gehrmann et al. Corrected cost ratio: ~10× vs GPT-4o (fully-loaded TCO), not 25×. The 25× figure in v1 compared only inference API costs against the GPU serving cost, ignoring storage, egress, retraining, and engineering labor. At 10M queries/day, a $14M annual API bill vs. a $400K fully-loaded self-hosting TCO is still a compelling 35× reduction on a pure-cost basis. However, engineering labor and operational risk are real costs that must be factored for any honest architectural recommendation. The break-even query volume for preferring SFT over API is approximately 300K–500K queries/day for a team with existing ML infrastructure.
